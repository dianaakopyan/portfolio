Context

DN.ru is one of Russia's largest suppliers of industrial pipeline valves — selling directly and through a network of regional dealers.

The old model was slow by design. A dealer orders goods → goods travel to dealer's warehouse → dealer ships to their client. Two logistics steps where one would do. Double handling, double storage, delays at every stage.

Schema: old model vs new model

The Idea

DN fulfills straight to the dealer's end clients. Cut out the middle step entirely.

DN earns on logistics margin, dealers save on storage and operations. No off-the-shelf WMS could support this. Complex industrial nomenclature, multi-location stock, dealer-level permissions, a fulfillment model that didn't exist yet in any product we evaluated.

So we built from scratch. My role: sole designer and product owner. Worked directly with the warehouse director, company owner, and the people who'd actually use it — warehouse staff across five locations. 12 months.

All screens are the DN operator side of the product.

My Warehouse — main screen

Main Screen

Managers were opening 3 tools to answer 1 question.

1C for stock. Excel for reservations. A phone call to confirm what was actually on the shelf. We watched this happen before touching anything. 11 context switches in a single 20-minute session. The data existed — it just lived in the wrong places.

One table replaced all three. Stock, reservations, incoming — all warehouses in parallel columns. Not tabs, because every tab switch broke concentration. We measured how often people switched before we cut the feature entirely.

Red numbers for overreserved stock are intentional. Early prototypes used subtle indicators — operators missed them completely. When you're moving fast, anything that doesn't demand attention is invisible.

Search screen

Search

Nobody was browsing by category. Everyone was typing a SKU.

We tracked navigation patterns across eight sessions before cutting the sidebar entirely. Operators know their product codes. They don't explore — they look up.

Search is the primary entry point. Results instant, item previewed before you commit. The action bar only appears on selection, only shows what's valid.

↳ Early builds showed a full toolbar at all times. Support logs filled with "I clicked it and nothing happened." Contextual visibility fixed it completely.
Item details popup

Item Details

Click a number. See everything behind it.

Customer, quantity, which warehouse is holding it, which document it's tied to.

That lookup used to average 4 minutes. Now it's under 30 seconds.

Column filter panel

Filters

Filters go beyond sort and hide. Negative stock, below-minimum threshold — both built directly into the column config.

↳ Both showed up as manual daily checks in nearly every observation session. We stopped treating them as edge cases.
Grouped view

Grouped View

Group by reservation status, then incoming status. The table becomes a live operational snapshot — aggregated totals per group, active config always visible, each condition removable.

Card sorting with the team showed a consistent pattern: managers think in status first. What's reserved, what's coming, what's sitting untouched. We designed grouping to mirror that directly.

Templates screen

Templates

Different roles configure the table differently. Templates save the full state — columns, grouping, sort, warehouses.

↳ 6–8 manual adjustments per session, every morning, before anyone started actual work. Templates ended that ritual.
Actions menu

Actions

Global operations live separately from item-level actions. New inventory count, CSV upload, save template — all in one place, never mixed with single-item actions.

Error logs from prototype testing showed the same mistake repeatedly: users triggered global operations thinking they were acting on a selected item. Structural separation — not just visual — eliminated that error class.

Delivery Flow

Three months of logs before touching the delivery flow.

Same two patterns every week: duplicate documents and stock discrepancies after supplier CSV imports. Not random. Systemic. Both traced back to specific moments in the delivery creation flow.

Before / after — error source diagram
Create new delivery screen

Create Delivery

The delivery form calculates as you build. Items by storage spot and package type. Weight, dimensions, costs — live. Receiving fee and storage shown before you commit, not discovered afterward.

↳ The by-package / by-item toggle wasn't planned. Came out of a mid-project walkthrough — half the team thought in pallets, the other half in individual items. Same data, two structures. We built both.
File import screen

File Import

Nothing gets imported silently.

Every CSV line validated before anything is committed. Errors flagged inline with a specific message. Downloadable as a list — fix externally and re-upload, or add manually.

Supplier files had always imported without validation. Problems surfaced days later after stock counts had already propagated incorrectly. One change eliminated the pattern entirely.

Duplicate items popup

Duplicates

Duplicate detected — system stops and asks. Merge or add to a new spot.

↳ No consensus among users on what should happen. About half expected a merge, half expected a new line. So we stopped guessing and made it an explicit choice.
Multi-select — contextual actions bar

Selection

Select items. The toolbar becomes a contextual action bar.

Actions depend on the selection — not a fixed list. Item count shown prominently at the left.

Operators kept losing track of their selection after scrolling. The count label was a late addition — one afternoon to build, immediate impact in the next testing session.

Manage stock popup

Stock Detail

Click any stock figure: total, reserved, incoming, available — plus direct actions tied to that item right there.

↳ "I know the number, I just don't know what's behind it." Came up in almost every interview.
Request inventory count popup

Inventory Count

Trigger a count from any item. Link created automatically. The manual reconciliation step that used to follow every count: gone.

Deliveries list

Deliveries

The delivery log is a working tool, not an archive.

Status shown as label and progress bar — state and progress readable at a glance. Delivery method as a colored tag. Line count and unit count in separate columns.

↳ Eye-tracking showed text labels weren't fast enough at row density. The bar gave operators something to scan, not just read. Line count and unit count were separated after analytics showed they drove different decisions — complexity vs. logistics planning.
